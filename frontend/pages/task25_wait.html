The wait() System Call

An interactive visual guide to process synchronization, zombie prevention, and exit status in Linux.

⏸️

Blocks Parent

The parent process suspends execution until one of its child processes finishes.

📤

Exit Status

Allows the parent to read the child's exit code using WIFEXITED and WEXITSTATUS.

🧹

Cleanup

The OS completely frees the child's resources after wait() is called.

🧟

Zombie Prevention

Without wait(), a terminated child lingers as a zombie in the process table.

Code Walkthrough

example_4_1_simple_wait.c C 
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();       // Create child process

    if (pid == 0) {            // ── CHILD process ──
        printf("Child executing...\n");
        sleep(2);               // Simulate 2s of work
        printf("Child finished.\n");
        exit(0);

    } else if (pid > 0) {      // ── PARENT process ──
        printf("Parent waiting for child...\n");
        wait(NULL);             // ← BLOCKS here until child exits
        printf("Parent: Child has been reaped.\n");
    }
    return 0;
}
example_4_2_wait_status.c C 
#include <stdio.h>
#include <stdlib.h>
#include <sys/wait.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {            // ── CHILD ──
        printf("Child processing data...\n");
        exit(42);               // Exit with code 42

    } else if (pid > 0) {      // ── PARENT ──
        int status;
        wait(&status);          // Pass pointer → capture exit info

        if (WIFEXITED(status)) {  // Did child exit normally?
            int code = WEXITSTATUS(status); // Extract the code
            printf("Exit code: %d\n", code);   // Prints: 42
        } else {
            printf("Child terminated abnormally.\n");
        }
    }
    return 0;
}
example_4_3_zombie.c C 
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int main() {
    pid_t pid = fork();

    if (pid > 0) {             // ── PARENT ──
        printf("Parent running. NOT calling wait()!\n");
        sleep(20);              // Parent alive but ignoring child
        printf("Parent exiting now.\n");

    } else if (pid == 0) {    // ── CHILD ──
        printf("Child exiting immediately.\n");
        exit(0);
        // ☠ Child is now ZOMBIE — entry stays in process table
        // because parent never called wait() to reap it.
    }
    return 0;
}

// Run:  ps aux | grep Z   to see [defunct] zombie entries
WIFEXITED(status)

Returns true if the child terminated normally via exit() or returning from main(). Returns false if killed by a signal.

WEXITSTATUS(status)

Extracts the actual exit code (0–255) passed to exit(). Only valid when WIFEXITED is true.

WIFSIGNALED(status)

True if the child was killed by a signal (e.g., SIGSEGV, SIGKILL). Use WTERMSIG(status) to get the signal number.

Key difference: wait(NULL) discards the exit status — use it when you don't care how the child exited, only that it did. Passing &status lets you inspect the exit code with macros like WIFEXITED and WEXITSTATUS. For waiting on a specific child, use waitpid(pid, &status, 0).

Common Mistakes

🧟
Forgetting wait()
Child exits but parent never calls wait(). The child becomes a zombie — its entry stays in the process table consuming resources until the parent exits.
Always call wait() or waitpid() after fork() to reap children.
📭
Ignoring Exit Codes
Using wait(NULL) discards the child's exit status. If the child fails with a non-zero code, the parent has no way to detect or handle the error.
Pass &status and use WIFEXITED / WEXITSTATUS to check for errors.
👻
Wrong PID Assumption
Both parent and child continue after fork(). Forgetting to check the return value means both processes execute the same branch, causing subtle bugs.
Always check: pid == 0 (child), pid > 0 (parent), pid < 0 (error).
🔒
Deadlock via wait()
If a parent calls wait() but has no children (because fork() failed), it blocks forever waiting for a process that will never come.
Check fork() return value. Only call wait() if pid > 0 succeeded.
📡
Missing SIGCHLD Handler
In long-running daemons, not handling SIGCHLD asynchronously causes zombies to accumulate over time, gradually exhausting the process table.
Set signal(SIGCHLD, SIG_IGN) or install a SIGCHLD handler that calls waitpid(-1, NULL, WNOHANG).
